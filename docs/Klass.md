# Unity & Unreal Runtime Class Field View: "Klass" vs. "Class"

This document provides a guide to understanding **Klass** and **Class** concepts in Breeze, how to use the **Class field** feature in the Memory Explorer, and how it behaves across Unity (IL2CPP) and Unreal Engine targets.

---

## 1. What is a Klass in Unity/IL2CPP?

In Unity IL2CPP games, every managed object instance in memory (e.g., custom scripts, game objects, components) has a pointer at offset `0x00` that points to its runtime descriptor structure, known as `Il2CppClass`. 

In IL2CPP internals (and within Breeze's source code), this structure and its pointer are referred to as the **Klass** (representing `Il2CppClass*`).

```
[Managed Object Instance in Heap]
Offset 0x00: Klass Pointer (Il2CppClass*) ──────> [Il2CppClass Struct (Klass)]
Offset 0x08: Monitor                              ├── Name: "PlayerController"
Offset 0x10: UnityEngine.Object fields            ├── Namespace: "Gameplay"
Offset 0x20: Instance Fields (e.g., m_Health)     ├── Parent Pointer (Base Class)
                                                  └── Fields Array (Metadata & Types)
```

The Klass is the engine's runtime descriptor. It tells the Unity engine—and Breeze—exactly how to interpret the raw bytes of that specific object instance. It contains:
- The class name and namespace (e.g., `Gameplay.PlayerController`).
- A pointer to the parent/base class (allowing walk up the inheritance chain to `MonoBehaviour` and `UnityEngine.Object`).
- A pointer to the runtime field metadata (`Il2CppFieldInfo`), which defines each field's name, type, and memory offset.

### How to Retrieve the Klass Name String

If you are inspecting memory manually or writing custom hook logic, you can retrieve the readable name of the class directly from the **Klass pointer** (`Il2CppClass*`):

* **Name String Pointer (`+0x10`)**: On standard 64-bit architectures (like the Nintendo Switch), the pointer to the null-terminated ASCII class name string is located at offset `+0x10` from the start of the Klass struct.
  - C++ Representation: `const char* name = *reinterpret_cast<const char**>(klass_ptr + 0x10);`
  - Dereferencing `[klass_ptr + 0x10]` yields the raw ASCII string (e.g., `"PlayerController"`).
* **Namespace String Pointer (`+0x18`)**: Similarly, the pointer to the null-terminated ASCII namespace string resides at offset `+0x18`.
  - Dereferencing `[klass_ptr + 0x18]` yields the namespace string (e.g., `"Gameplay"`).

## 2. How is Klass Different from Class?

While they sound identical, they refer to different stages and structures in the development and reverse engineering lifecycle:

| Feature | Class (C# / Design-Time) | Klass (C++ / Runtime Struct) |
| :--- | :--- | :--- |
| **Context** | C# code declaration and static metadata. | Live C++ memory descriptor (`Il2CppClass`). |
| **Source** | Defined by developers in code; indexed in `dump.cs`. | Generated by the IL2CPP compiler and loaded in game RAM. |
| **Location** | Static database, text files (`dump.cs`). | Dynamic memory heap (allocated by the game process). |
| **Instance Relationship** | A blueprint. No direct instance state. | Every object instance points to its Klass at offset `0x00`. |
| **Functionality** | Declares *what can exist* (methods, fields, offsets). | Allows Breeze to *dynamically discover* and parse live fields at runtime. |

In short, a **Class** is the static blueprint you read in a code editor or `dump.cs`, whereas a **Klass** is the active C++ descriptor structure in the Switch's memory that Breeze queries to resolve fields for a live object.

---

## 3. Basics of a Class in Reverse Engineering

When reverse engineering a game, classes represent the roadmap to modifying game state:
- **Fields**: Raw variables stored in memory.
  - **Instance Fields**: Stored inside each object instance (e.g., a specific player's `health` or `coordinates`). Their addresses are relative to the object's base address (`instance_base + offset`).
  - **Static Fields**: Variables shared across all instances of a class. They are stored globally at a dedicated class-level static base address, rather than inside individual objects.
- **Offsets**: The exact distance (in bytes) from the start of the object to a specific field. For example, if a `Player` object is at `0x40000000` and the `health` field offset is `0x2C`, the health value resides at `0x4000002C`.
- **Inheritance**: Derived classes inherit fields from their parent classes. For instance, a `PlayerController` inherits fields from `MonoBehaviour`, which inherits from `Behaviour`, `Component`, and `UnityEngine.Object`.

## 4. How to Use the "Class field" Button in Memory Explorer

Breeze allows you to transition instantly from raw memory inspection to high-level object fields.

### Step-by-Step Usage:
1. **Navigate to an Address**: In the **Memory Explorer**, find a memory address that you suspect points to a class instance (such as a Player, Enemy, or Manager object pointer).
2. **Press the Shortcut**: Press the **Class field** button (shortcut: `Y + ZR + ZL`).
3. **Automatic Resolution**: 
   - Breeze reads the pointer at offset `0x00` of your selected address (the suspected Klass pointer).
   - It validates the Klass self-pointer (`[klass+0x78] == klass` for non-arrays).
   - It heuristically parses the layout of the `Il2CppClass` structure based on the game's Unity metadata version (legacy, mid, or modern profiles).
   - It opens the **Field View** for the resolved class.
4. **Pin the Instance**: Pin the base address using the `Pin selected offset` action to view live, auto-updating values.

## 5. Unreal Engine Support: "Class field" Differences & Limitations

The **Class field** button is also available when inspecting Unreal Engine games (shortcut `Y + ZR + ZL` in Memory Explorer or `Y + ZR` in the Gen2 captured rows menu). However, the Unreal implementation is **less powerful and more restrictive** than the Unity IL2CPP equivalent.

### Why it is less powerful for Unreal:

1. **Stripped Reflection Metadata**:
   - **Unity (IL2CPP)**: Preserves complete metadata for every C# class, method, and field in `global-metadata.dat`. Breeze can always reconstruct the complete layout, variable names, and types.
   - **Unreal Engine**: Shipping builds often strip extensive reflection metadata. Only properties explicitly marked as `UPROPERTY` or utilized by blueprint/network replication systems remain in the runtime binary. Many internal C++ member variables will not show up in the runtime field chain because their metadata does not exist in the shipping executable.
2. **Large Object Offsets**:
   - Unreal Engine objects (`UObject`) can be extremely large, with fields residing at offsets exceeding `0x3000` bytes from the object header.
3. **Automatic Base Recovery Failures**:
   - When you press **Class field** in the Memory Explorer, Breeze attempts to scan backward to find the UObject header (`ClassPrivate`, `NamePrivate`, `OuterPrivate`).
   - Because the probe window is limited, if your cursor is positioned deep inside a large object (far away from the header), Breeze's automatic base recovery will fail.
   - **Workaround**: Move your cursor closer to the suspected UObject base pointer (like a pawn or controller pointer) before pressing **Class field**, or prefer opening it via the `Gen2 Menu -> Class field` which carries stronger context.

## 6. What You See in the Klass Field View

Once the Field View is successfully resolved, Breeze displays a structured table containing the following columns:

* **Offset**: The hex offset of the field relative to the object base (e.g., `0x18`, `0x2C`).
* **Type**: The parsed C# type name (e.g., `int32`, `float`, `bool`, `System.String`, or custom types/enums).
* **Name**: The C# variable name as defined by the developers (e.g., `m_Health`, `goldCount`, `playerName`).
* **Address**: The live absolute memory address where the field resides.
* **Value**: The current value stored at that address, updated in real time if the base address is pinned.

### Advanced Features in the Field View:
- **Expandable Collections**: Arrays and Lists can be expanded to inspect their elements.
- **String Decoding**: `System.String` types are automatically dereferenced and displayed as readable text (e.g., `"Player_1"`).
- **Static Fields**: Displayed in their own section, representing global class-level variables rather than instance-level values.
- **Relationship Navigation**:
  - **Class Link**: Instantly searches for other classes containing fields that link to the current class.
  - **Descendent**: Displays classes that inherit from the current class.
- **Handoff & Automation**:
  - Jump back and forth between the Field View, Memory Explorer, and raw `dump.cs` while keeping pointer-chain context intact.
  - Generate cheat codes automatically (`Make cheat`) for selected fields if a stable pointer chain is established.

## 7. Distinguishing Friends vs. Foes Using Klass (Inheritance)

A common challenge in game hacking is targeting *only* enemies (e.g., One-Hit Kill) or *only* players/allies (e.g., Infinite Health) when the game uses a shared class structure for both.

### The Problem: Shared Class Methods
Suppose the game has a base class called `Character` or `Actor` containing a health field and a damage method:

```csharp
public class Actor : MonoBehaviour
{
    public float currentHealth;
    
    public void TakeDamage(float amount)
    {
        this.currentHealth -= amount;
    }
}
```

Both your player/allies (friends) and enemies (foes) will run the exact same `TakeDamage` method when hit. If you hook `Actor::TakeDamage` to disable damage, **both friends and foes will become invincible**.

### How Inheritance Works
In Object-Oriented Programming (and Unity), we use inheritance to specialize behavior. Instead of using `Actor` directly, developers create subclasses:

```csharp
public class Player : Actor { /* Player-specific logic */ }
public class Enemy : Actor  { /* Enemy-specific logic */ }
```

- `Player` inherits from `Actor` (acquiring the `currentHealth` field and `TakeDamage` method).
- `Enemy` inherits from `Actor` (also acquiring `currentHealth` and `TakeDamage`).

### The Solution: Klass as a Sure Identifier
At runtime, even though the CPU executes the *same* shared code block for `Actor::TakeDamage`, the `this` pointer points to different physical instances in memory. 

By checking the **Klass** pointer (at offset `0x00`) of the active instance, you can reliably distinguish friend from foe:

1. **Player Instance (Friend)**:
   - Object base address: `0x45001200`
   - Pointer at `[0x45001200 + 0x00]` points to the **`Player` Klass** (`Player_Il2CppClass*`).
2. **Enemy Instance (Foe)**:
   - Object base address: `0x45089700`
   - Pointer at `[0x45089700 + 0x00]` points to the **`Enemy` Klass** (`Enemy_Il2CppClass*`).

```
Inside Actor::TakeDamage(this_ptr, amount):
  
                       ┌─── [this_ptr + 0x00] == Player_Klass? ───► YES (Friend) ──► Skip Damage!
                       │
  Inspect Klass ───────┤
                       │
                       └─── [this_ptr + 0x00] == Enemy_Klass?  ───► YES (Foe)    ──► Apply 9999 Damage!
```

By comparing the instance's Klass pointer at `0x00` against the known Klass address of `Player` or `Enemy`, you can write precise conditionals in your cheat scripts (such as ASM cheats or memory pokes) to selectively apply modifications only to the intended target.
